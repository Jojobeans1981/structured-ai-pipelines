In a Kitchen...

📋 A recipe arrives
👨‍🍳 Chef reads it, writes a prep list
🔪 Chop veggies (can't cook unchopped food)
🍳 Cook in the right order
👅 Taste test before serving

In AI Factory...

📄 A feature contract arrives
🤖 Plan generator reads it, creates stage DAG
🗄️ Generate database migration first
⚡ Generate API, frontend, tests in order
✅ Run linter + tests before anyone sees it
📦 Package as a merge request

3. What Every Stage Receives Before It Starts

No stage runs blind. Every stage gets these four things:

📄
The Contract
The original feature request. What are we building?
🔍
Repo Intelligence
What does the codebase already look like? What patterns does it use?
📦
Prior Artifacts
What did the previous stages produce? The API stage sees the migration output.
🛡️
Governance Rules
What are we NOT allowed to do? Security policies, style rules.
4. Save Points (Why Failures Don't Erase Progress)

Think of it like a video game. After every stage, the game saves. If you die at stage 4, you respawn at stage 4 — not at the beginning.

7. How I Actually Build Each Step
P
Plan Generator
Reads the feature contract and creates the dependency graph.
Input: "Add user authentication with email/password"

Output:
  migration → no dependencies
  config → no dependencies
  api → depends on [migration, config]
  frontend → depends on [api]
  tests → depends on [api, frontend]
  verify → depends on [tests]
  package → depends on [verify]
How: An LLM (routed via the model router — Groq free tier or Claude) reads the contract, identifies what artifact types are needed, and outputs a JSON dependency graph. The DAGExecutor validates it for cycles and missing references before any stage runs.
1
Migration Stage
No dependencies — can start immediately
Generates database schema changes for the feature.
Receives: contract + repo intelligence (existing schema, ORM type)
Produces: SQL migration file or Prisma schema diff
Example: CREATE TABLE users (id UUID, email VARCHAR UNIQUE, password_hash VARCHAR, ...)
How: The stage executor loads the migration-generator skill prompt. It passes the contract + detected ORM (Prisma, Knex, raw SQL) to the LLM. The LLM generates a migration file that matches the repo's existing migration style. Output is saved as an artifact by the control plane.
2
Config Stage
No dependencies — runs parallel with Migration
Generates environment variables, service wiring, and config files.
Receives: contract + repo intelligence (existing .env, config pattern)
Produces: .env additions, config module updates
Example: JWT_SECRET=, SESSION_EXPIRY=3600, AUTH_PROVIDER=local
How: Loads the config-generator skill. Reads the existing config pattern (dotenv, JSON config, YAML). Generates only the NEW config entries the feature needs. Doesn't touch existing values.
3
API Stage
Waits for Migration + Config
Generates route handlers, controllers, and service layer.
Receives: contract + migration artifact + config artifact + repo conventions
Produces: Route files, service classes, middleware
Example: POST /api/auth/login, POST /api/auth/signup, POST /api/auth/logout
How: This is where prior artifacts matter. The API stage SEES the migration output — it knows the table structure, column names, types. It generates routes that match the actual schema, not a guess. It also sees the config output so it uses the right env vars. Routed to Claude (heavy task — needs complex reasoning about API design).
4
Frontend Stage
Waits for API
Generates UI components, pages, and client-side logic.
Receives: contract + API artifact (knows the exact endpoints) + repo conventions (React? Vue? Svelte?)
Produces: Page components, form handlers, API client calls
Example: LoginPage.tsx, SignupForm.tsx, useAuth.ts hook
How: The frontend stage reads the API artifact to know the exact endpoints, request/response shapes, and auth flow. It generates components that call the REAL API — not placeholder URLs. It matches the repo's component style (functional vs class, CSS modules vs Tailwind, etc.) because repo intelligence told it what to match.
5
Test Stage
Waits for API + Frontend
Generates unit and integration tests against the generated code.
Receives: ALL prior artifacts (migration + config + API + frontend)
Produces: Test files using the repo's test framework
Example: auth.test.ts — tests login success, login failure, signup validation, logout
How: Detects the repo's test framework (Jest, Vitest, Pytest, Go test). Generates tests that import the ACTUAL generated code — not mocked abstractions. Tests cover happy path, error cases, and edge cases. Routed to Claude (heavy — writing good tests requires understanding the full feature).
6
Verify Stage
THE QUALITY GATE
Runs the repo's own lint command and test command. Not ours. THEIRS.
Receives: All generated files written to disk
Runs: npm run lint + npm run test (or whatever the repo uses)
If FAIL: Error output is captured. Stage goes to awaiting_approval. Human can reject to retry.
If PASS: Proceeds to packaging.
How: NO LLM involved. This is pure execution. The build-verifier detects the project type (package.json = Node, pyproject.toml = Python, go.mod = Go), finds the lint and test commands, runs them with a timeout (60s lint, 120s tests), captures stdout/stderr, and reports pass/fail. If the generated code doesn't pass the repo's own quality gates, we catch it HERE — before any human reviews it.
7
Package as Merge Request
ONLY runs if verify passed
Bundles all generated files into a single change set and opens a merge request.
Collects: migration file + config changes + API routes + frontend components + test files
Creates: Git branch, commits files, opens GitLab MR with generated title + description
Output: MR URL — ready for human code review
How: The PR assembler collects every artifact from every stage, creates a feature branch (feat/auth-{timestamp}), commits the files, and uses the GitLab API to open a merge request. The MR description references the original feature contract and lists every file generated with a one-line summary. A human reviews it like any other MR. The AI proposed. The human disposes.

Feature contract goes in.
Seven stages run in dependency order.
Every output is saved.
